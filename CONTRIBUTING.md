# Contributing Guide

이 프로젝트는 사람과 AI가 함께 작업하는 저장소입니다. 새 기능을 추가하거나 기존 기능을 바꿀 때는 아래 규칙을 기본값으로 사용합니다.

## 기본 원칙

- 코드만 바꾸지 말고 문서, 에러 코드, 버전, 테스트까지 함께 관리합니다.
- 백엔드와 프론트의 계약은 항상 동시에 맞춥니다.
- 사용자에게 보이는 설명은 한국어 우선으로 정리합니다.
- 예측 기능은 과장보다 검증 가능성과 일관성을 우선합니다.

## 버전 정책

- `PATCH`
  - 버그 수정
  - 문서 보강
  - 에러 코드 정리
  - UI 문구 개선
- `MINOR`
  - 새로운 사용자 기능
  - 새 API 라우터
  - 예측 엔진 개선
  - 화면 구조 확장
- `MAJOR`
  - 호환성 깨짐
  - 기존 API 계약 변경
  - 핵심 데이터 모델 변경

버전을 변경할 때는 아래 파일을 함께 맞춰주세요.

- `backend/app/version.py`
- `frontend/package.json`
- `frontend/package-lock.json`
- `CHANGELOG.md`

## 변경 시 반드시 확인할 항목

### 1. README

아래 중 하나라도 해당되면 `README.md`를 업데이트합니다.

- 사용자 기능이 추가되거나 제거됨
- 실행 방법, 환경 변수, 페이지 구성, API 사용법이 달라짐
- 예측 엔진 설명이나 데이터 소스가 바뀜
- 다운로드/아카이브/캘린더처럼 눈에 띄는 UX가 바뀜

### 2. 에러 코드

아래에 해당하면 `backend/app/errors.py`와 `README.md`의 에러 코드 표를 같이 갱신합니다.

- 새로운 `AppError`를 추가함
- 기존 에러 코드의 의미가 달라짐
- 새로운 장애 유형을 라우터에서 직접 처리함

### 3. 프론트-백엔드 계약

아래에 해당하면 프론트 타입과 소비 코드를 같이 업데이트합니다.

- 응답 JSON 필드 추가/삭제/이름 변경
- 날짜 형식, 숫자 단위, 상태 문자열 변경
- 내보내기나 다운로드 경로 변경

대표 점검 파일:

- `frontend/src/lib/api.ts`
- 관련 페이지와 컴포넌트

### 4. 예측 엔진

예측 로직을 바꾸면 다음도 같이 반영합니다.

- 모델 버전 문자열
- 시스템 진단 또는 모델 설명 문구
- `README.md`의 예측 로직 설명
- `CHANGELOG.md`
- 최소 1개 이상의 회귀 테스트

### 5. 한국어 사용자 문구

다음은 한국어 우선으로 유지합니다.

- 카드 설명
- 상세 패널
- 운영 상태 문구
- 아카이브/다운로드 안내
- 분석 해설

영어 유지가 자연스러운 경우:

- 티커
- 고유 API 이름
- 국제 지표 약어
- 외부 서비스 이름

## Definition of Done

머지 가능한 상태는 아래를 충족해야 합니다.

- [ ] 기능 코드가 의도대로 동작함
- [ ] `README.md`가 최신 상태임
- [ ] `CHANGELOG.md`가 최신 상태임
- [ ] 필요한 에러 코드가 등록됨
- [ ] 백엔드/프론트 버전이 동기화됨
- [ ] 프론트 타입과 API 계약이 일치함
- [ ] 사용자 문구가 한국어 중심으로 정리됨
- [ ] 테스트 또는 회귀 검증이 존재함
- [ ] 아래 명령 검증이 완료됨

## 권장 검증 명령

### Backend

```powershell
cd backend
& ..\venv\Scripts\python.exe -m compileall app
& ..\venv\Scripts\python.exe -m unittest discover -s tests -v
```

### Frontend

```powershell
cd frontend
npm run build
npx tsc --noEmit
```

## 커밋과 PR 작성 팁

- 커밋 메시지는 무엇을 바꿨는지 한 줄로 드러나게 작성합니다.
- PR 본문에는 사용자 영향, 테스트 결과, 문서 반영 여부를 함께 적습니다.
- 예측 엔진이나 데이터 소스를 바꿨다면 근거와 제한 사항을 남깁니다.
